# Sekkei Specification — v1.1

**Status:** validated against two sekkeis: `kizo:food.fullservicerestaurant @ A.0` (reverse-engineered from the BaanBaan POS, ~170 nodes) and `kizo:web.todomvc @ A.0` (forward-designed from the canonical TodoMVC behavior, 57 nodes).
**Companion artifacts:**
- `specification/sekkei_schema.yaml` — informative schema (this document is its narrative).
- `specification/sekkei.schema.json` — normative JSON Schema, machine-checkable.
- `compass_artifact_…_text_markdown.md` — the GLM essay this spec implements (§C "Formal structure of a sekkei").
**Audience:** humans who author or read sekkei nodes by hand, and the regenerators (LLM agents + tooling) that consume them.

This document is a specification, not a tutorial. It says what a sekkei *is*; for the *why*, see the GLM essay.

---

## 1. Purpose and scope

A **sekkei** (設計) is a directed acyclic graph of typed, content-addressed nodes that captures the design of a software system intended to be generated by an LLM agent rather than written by hand. The sekkei is the source artifact under management; generated code is a derived build product.

This specification defines:

1. The five strata that nodes belong to (`system`, `capability`, `component`, `interaction`, `spec`) plus the §C.1 amendment permitting System recursion (Sub-Systems).
2. The common envelope every node carries (id, stratum, title, revision, provenance, parameters, constraints, relationships, body).
3. The stratum-specific `body` shape, including the six `spec_kind` variants for spec leaves.
4. The four inheritance operations (`as_is`, `with_override`, `extend`, `net_new`) and the locality rule §C.2.
5. The relationship taxonomy (`composes-of`, `depends-on`, `derives-from`, `implements`, `generates`, `varies-from`).
6. The file-system layout convention validated by both sekkeis.
7. The verification gates a sekkei must pass before it is considered authored.
8. The authoring conventions that emerged from validation but are not enforced by the schema.

The dBOM, sekkei.lock, the seven GLM processes (Change Management, Variant Resolution, etc.), and the two-dimensional cache are specified separately in the GLM essay (§C.6, §D, §E) and are out of scope for this document.

## 2. Strata

A node's `stratum` field places it in one of five layers. The hierarchy is strict: the only `composes-of` edges allowed are those declared in the table below. The verifier (§7) enforces this.

| Stratum       | Contains (composes-of) | PLM analog                | Cardinality              |
|---------------|------------------------|---------------------------|--------------------------|
| `system`      | `system`, `capability` | End-Item / Configured Engineering Item | 1 root + N Sub-Systems |
| `capability`  | `component`, `interaction`, `spec` | Major sub-assembly      | 1..N per System         |
| `component`   | `interaction`, `spec`  | Assembly                  | 1..N per Capability     |
| `interaction` | `spec`                 | Sub-assembly / interface  | 1..N per Component       |
| `spec`        | (leaf)                 | Drawing / part specification | 1..N per Interaction    |

**§C.1 amendment — System recursion.** Originally the strata were strict (System could not contain System). Reverse-engineering BaanBaan revealed a real design pressure: the root System had four large internal divisions (`register`, `communication`, `onlineordering`, `reservations`) that needed System-level structuring (own parameters, own dBOM-eventually, own Capabilities) without being separate deployable products. The amendment permits a System to compose other Systems, called **Sub-Systems**. The strict hierarchy below the System tier is preserved; only the System tier is recursive. A Sub-System is identified by a `kizo:` id that extends its parent's id (e.g., `kizo:food.fullservicerestaurant.register` is a Sub-System of `kizo:food.fullservicerestaurant`).

**§C.1.b amendment — `system_role` discriminator (v1.1.9).** A root System and a Sub-System share the same structural shape (both compose Capabilities, both have parameters, both have provenance and revisions), but they differ in *lifecycle role*: a root is the deliverable that ships to an operator; a Sub-System is a planning grouping that never leaves the design. PLM expresses this with a discriminator field on a single Item class, not with two separate classes (Aras: Top-Level boolean; Windchill: End-Item Type; Teamcenter: Item Type taxonomy). The sekkei schema follows the same pattern: every System node declares `body.system_role`, with values:

- **`root`** — the deliverable product family unit. Cardinality: exactly 1 per sekkei graph. MUST own an `acceptance_gate`. MUST have `dbom_ref` (a path, or explicit `null` with documented rationale at small scale).
- **`subsystem`** — a structural grouping under a root or another Sub-System. Cardinality: N. MUST have `dbom_ref: null` (Sub-Systems do not deploy independently). Inherits the root's `acceptance_gate`.
- **`platform`** — *reserved.* A shared-across-products structure whose parameter defaults are inherited by descendant variants (e.g., VW MQB-style). May itself have a `dbom` if it independently deploys. Not yet realized in any sekkei.

The discriminator is checked at evaluation time by the verifier (gate 2.b — role consistency). The schema enforces conditional rules: `root → required acceptance_gate`; `subsystem → dbom_ref must be null`.

A spec is a **leaf**. It does not compose. If a spec wants to reference another spec (e.g., a schema spec referenced from an acceptance spec), it does so by `$ref`-style id reference inside its body, not by `composes-of`.

## 3. The common envelope

Every node, regardless of stratum, carries the following fields. The five required fields (id, stratum, title, revision, provenance) are checked by gate 1 of the verifier.

```yaml
id: kizo:web.todomvc.todo_management.todo_repository
stratum: component
title: Todo Repository
description: |
  Owns the todos table and the CRUD operations against it...
content_hash: <computed at evaluation time; not stored in source>

revision:
  major: A                  # A..Z (Y14.35: skip I, O, Q, S, X, Z)
  iteration: 0              # increments on every in-work edit
  status: in_review         # in_work | in_review | released | superseded | obsolete

provenance:
  derives_from: null        # or { id, content_hash } if inherited
  override_kind: net_new    # as_is | with_override | extend | net_new
  authored_by: glm-sekkei-validator@A.0
  authored_at: 2026-05-10T00:00:00Z
  generator_identity: null  # only on nodes produced by a generator

parameters: [...]           # see §5
constraints: [...]          # see §6
relationships: [...]        # see §4
body: {...}                 # stratum-specific; see §8
```

### 3.1 Identifier rules

- The `id` is a slugified URI. Lowercase, dot-separated path under the `kizo:` prefix.
- The id of a child node MUST be prefixed by the id of its `composes-of` parent. (The verifier does not enforce this strictly — domain-specific shortenings are permitted — but every sekkei authored to date has obeyed it.)
- An id is the **logical** identifier. Two nodes with the same id but different content hashes are different revisions of the same configuration thread.

### 3.2 Revision

Major letters follow ASME Y14.35: A, B, C, D, E, F, G, H, J, K, L, M, N, P, R, T, U, V, W, Y. (Skip I, O, Q to avoid character confusion; skip S because it's reserved for "superseded" status; skip X because it's reserved for experimental drops; skip Z to leave headroom.)

`iteration` increments on every in-work edit and resets to 0 on a major release. A release is when `status` advances to `released`, after which the node is immutable for that (major, iteration). A new edit increments to a new iteration in `in_work`.

### 3.3 Provenance

Every node MUST carry a `provenance` block. The four operations on `override_kind` are formally specified in §4. `authored_by` is an agent identity (e.g., `glm-reverse-engineer@A.0`) or a person. `authored_at` is ISO-8601 UTC.

`generator_identity` is the special-case block attached to nodes that were *produced by a generator* rather than authored. It pins the LLM identity, the prompt content hash, and the generator binary content hash, so that regeneration determinism can be reasoned about.

### 3.4 content_hash

The `content_hash` is a SHA-256 over the canonical YAML serialization of the node body PLUS the recursive content hashes of every node directly referenced via `composes-of` and `depends-on`. It is computed at evaluation time and is NOT written into the source files. It is the unit of identity for the design cache (essay §E.1).

## 4. Inheritance operations

A child sekkei inherits from a parent via four operations on each inherited node, declared via the `provenance.override_kind` field:

| Operation             | Semantics                                                                                                |
|-----------------------|----------------------------------------------------------------------------------------------------------|
| `as_is`               | Child references the parent node by `(id, content_hash)`. Mutations to the parent flow through per the child's `pin_policy`. |
| `with_override`       | Child references the parent and additionally specifies a JSON-Patch / RFC-6902 delta. The effective node is `merge(parent.body, child.overrides)`. |
| `extend`              | Child adds wholly new child nodes under an inherited parent. Used when a fork needs additional structure. |
| `net_new`             | The node is not present in any ancestor. Net-new is the principal locus of variant differentiation.       |

### 4.1 The locality rule (§C.2)

Inheritance is **local to the addressed node**. A `with_override` on a Component does NOT cascade into that Component's child Interactions or Specs unless those children also declare `with_override` referencing the corresponding parent. A child that wants to override a deep leaf MUST address it directly — there is no implicit parameter-driven structural cascade.

This is a deliberate departure from SPLE feature-model thinking. Parameter VALUES propagate (§5); structural OVERRIDES do not. The reason: structural cascades produce unbounded fan-out under the 80% rule, defeating the purpose of locality and making a sekkei's effective shape dependent on resolution order. Overrides at the leaf are visible at the leaf.

### 4.2 The 80% rule

The empirical platform-engineering target is that a fork inherits ~80% of the parent's structure as-is, with the remaining 20% in `with_override` + `extend` + `net_new`. The verifier does not enforce this number, but a fork whose `as_is` ratio is far below 80% is a signal that the parent should be split or the child should be recast as a sibling System.

## 5. Parameters

Parameters are typed configuration points declared on any node, with a `binding_scope` that determines how widely the parameter is visible.

```yaml
parameters:
  - name: server_port
    schema: { type: integer, minimum: 1024, maximum: 65535 }
    default: 3000
    binding_scope: system
```

### 5.1 Propagation

Parameters declared at higher strata are visible to all descendants. Parameters declared lower are local. A child MAY shadow a parent parameter to narrow its type or enum.

A **variant** of a sekkei is the tuple `(root_id, parameter_binding)` where the binding is a complete assignment of values to all parameters in the parameter set along the path from root to leaf.

### 5.2 Constraint checking

Constraints (`requires`, `excludes`, `invariant`, `acceptance`) are evaluated lazily at variant resolution time — never eagerly across the cross-product of all valid parameter assignments. This avoids the SPLE 10³⁰⁰⁰-configuration-space failure mode while preserving explicit `requires`/`excludes` invariants.

## 6. Relationships

Six typed relationships connect nodes. Each is a first-class object with an optional `attributes` map.

| Relationship    | Source → Target                        | Notes                                                       |
|-----------------|----------------------------------------|-------------------------------------------------------------|
| `composes-of`   | parent → child stratum node            | Structural parent-child within a sekkei tree. Carries `find_number`. |
| `depends-on`    | any → external dep (`pkg:` / `dep:` / `svc:` / `hw:`) | External artifact reused as-is. Carries version pin and digest. |
| `derives-from`  | child → parent@revision                | The inheritance link, with `override_kind` distinguishing as-is vs override vs extend. |
| `implements`    | Component or Interaction → Spec        | Connects an implementation node to the Spec(s) it must satisfy. |
| `generates`     | sekkei node → generated artifact id    | Provenance link from a node to a generated file. Carries the generation hash. |
| `varies-from`   | spec → spec                            | Horizontal alternates / substitutes (PLM Alternate vs Substitute distinction). |

### 6.1 External references

`depends-on` targets that are external use one of four prefixes:

- `pkg:` — a Package URL (PURL) for a published library, e.g., `pkg:npm/hono@4`
- `dep:` — an internal dependency that is not yet PURL-addressable
- `svc:` — a third-party hosted service, e.g., `svc:finix/api`
- `hw:` — a hardware target, e.g., `hw:raspberry_pi_5`

The verifier treats anything matching these prefixes as the closure boundary — it does not require an authored sekkei node for them, but it DOES require their pins to live in `nodes/dependencies/external_deps.yaml`.

## 7. Body shapes

The `body` field is stratum-specific. The schema (`sekkei.schema.json`) enforces the System / Capability / Component / Interaction shapes loosely (additionalProperties allowed, only `runtime` and `contract_kind` strictly required for their respective strata) and the spec.acceptance and spec.prompt shapes strictly.

### 7.1 System body

```yaml
# Root System (cardinality 1 per graph)
body:
  system_role: root                 # v1.1.9: required discriminator
  dbom_ref: ./dbom.yaml             # path to the dBOM (or null at small scale with rationale)
  realization_summary: |
    Single Bun process. Hono router. bun:sqlite (WAL)...
  acceptance_gate: |
    Regenerating from this sekkei must produce a system that:
      1. Passes every spec.acceptance.deliverable
      2. ...
```

```yaml
# Sub-System
body:
  system_role: subsystem            # v1.1.9: required discriminator
  dbom_ref: null                    # MUST be null — Sub-Systems do not deploy independently
  realization_summary: |
    Internal grouping of register-side Components: order_management,
    auth-scoped routes, dashboard surfaces...
  # acceptance_gate omitted — inherited from the root System
```

### 7.2 Capability body

```yaml
body:
  user_value: |
    Customers and staff can place, modify, pay, and complete a meal order.
  boundary: |
    Owns the order lifecycle, payment workflow, and staff dashboard.
    Does NOT own kitchen ticket printing (Kitchen Communication Capability).
```

### 7.3 Component body

```yaml
body:
  boundary: |
    Owns the todos table DDL and the per-operation SQL.
    Does NOT own: REST routing, filter semantics, or the SQLite connection.
  runtime: in_process              # or sidecar | external_service | in_browser | in_process_and_in_browser
  realization_file: src/repository.ts
  realization_files:               # alternative: multiple files
    - src/repository.ts
    - src/db.ts
  realization_notes: |
    The decrement reactor described here is PLANNED — the v2 codebase does
    not currently realize it. Documented for forks that want it.
```

The `boundary` field MUST state both what the Component owns AND what it does NOT own. This is the convention captured in §10 and is the single most-violated rule during reverse-engineering — Components that claim too much become impossible to regenerate independently.

### 7.4 Interaction body

```yaml
body:
  contract_kind: fsm               # or event_flow | integration_adapter | schema_binding
  contract_definition:
    pc: status                     # name of the prop-control field
    pc0: received                  # initial state
    states:
      - { id: received, terminal: false, transitions: [SUBMIT_TO_POS, CANCEL] }
      - { id: submitted, terminal: false, transitions: [POS_CONFIRMED, POS_ERROR] }
      # ... read VERBATIM from the realization file
    actions:
      SUBMIT_TO_POS: [submitted]
      # ...
    naps:
      placement: |
        Manually wired inside component.naps POST-INIT, NOT in state specs.
      principal_naps:
        - { state: received, fires: SUBMIT_TO_POS, when: instance_creation }
    reactors:
      - { name: dehydrate_to_sqlite, on: every_transition, target: orders_table_update }
    invariants:
      - "status_history is monotonic over the lifetime of one order"
  realization_file: v2/src/workflows/order-relay.ts
```

For `contract_kind: fsm`: the rule of §10 — **read the FSM source verbatim, never extrapolate** — is load-bearing. State names, action names, and NAP wiring are part of the contract the regeneration must reproduce; inferring "what the FSM probably looks like" from the domain concept produced rewrites of five files in the BaanBaan sekkei.

For `contract_kind: schema_binding`: contract_definition holds the DDL text, PRAGMAs, and indexes.

For `contract_kind: integration_adapter`: contract_definition holds request/response shapes and an error taxonomy.

For `contract_kind: event_flow`: contract_definition holds trigger, ordered steps, and end states.

### 7.5 Spec body — six `spec_kind` variants

`spec_kind` is a TOP-LEVEL field on spec nodes (v1.1 change from the §C.8 draft, where it was nested inside body). The verifier checks both that it is present and that its value is in the enum.

#### 7.5.1 `functional`

What the thing does, from the user/caller's perspective. Three alternative body shapes are supported; a node uses whichever fits its target:

- `behaviors[]` — for code modules. Each entry has `id`, `signature`, `description`, `preconditions[]`, `postconditions[]`.
- `endpoints[]` — for HTTP APIs. Each entry has `method`, `path`, `query`, `body`, `success`, `errors[]`.
- `responsibilities[]` + `page_structure` — for view-shell components.

#### 7.5.2 `technical`

How the thing is built. Free-form `implementation` block plus `module_layout`. Conventions:

- For backend modules: `runtime`, `storage`, `connection`, `prepared_statements`, `id_generation`, `timestamps`, `error_taxonomy`.
- For HTTP servers: `framework`, `bootstrap_layout`, `middleware`, `validation`, `error_translation`, `static_serving`.
- For frontend modules: `module`, `selectors_owned`, `event_listener`, `http_call`, `loading_state`.

#### 7.5.3 `schema`

The data shape. `data_shapes` (singular entity definitions) or `shapes` (request/response bundles). Cross-references via `ddl_reference: <interaction-id>` or `$ref: <spec-id>#/body/data_shapes/<Type>`.

#### 7.5.4 `business_rule`

Rules with enforcement traceability:

```yaml
body:
  rules:
    - id: BR-API-003
      rule: "DELETE on a missing id is 404, NOT 204."
      enforcement: route reads removeReturned and translates false → 404
```

The `id` field follows the convention `BR-<scope>-<NNN>` so rules are quotable in test names and code comments.

#### 7.5.5 `acceptance` — the deliverable shape

This is the most schema-disciplined spec_kind. The body MUST contain `deliverables[]` and `verifier`. The original §C.8 draft permitted free `inspection_assertions` prose; reverse-engineering showed that prose acceptance criteria do not survive contact with regeneration. The deliverable shape is:

```yaml
body:
  deliverables:
    - kind: test_file
      path: test/repository.test.ts
      framework: bun:test
      cases:
        - "create returns a Todo with completed=false and a non-empty id"
        - "list({ filter: 'active' }) excludes completed rows"
        - # ... one human-readable test description per case
    - kind: migration_file
      path: src/migrations/001_create_todos.sql
      content_reference: kizo:web.todomvc.todo_management.todo_repository.todo_schema
    - kind: fsm_acceptance
      reference: kizo:web.todomvc.web_ui.todo_list_view.edit_mode_fsm
      assertion: "every transition has at least one Playwright case exercising it"
  runtime_invariants:
    - "after any successful create/update, get(id) returns the same row"
  verifier:
    command: "bun test test/repository.test.ts"
    expect: "all tests pass; exit code 0"
```

Four `kind` values are recognized:
- `test_file` — a test suite with a path, framework, and case list.
- `migration_file` — a DDL or migration file with a content reference to the source-of-truth Interaction.
- `fixture_file` — seed data.
- `fsm_acceptance` — a coverage assertion against an FSM Interaction.

The `verifier.command` is a SINGLE shell line. Returning exit 0 is success. The spec author MUST write a command that is actually runnable in the regenerated workspace.

#### 7.5.6 `prompt` — the machine-runnable triple

A prompt spec is sufficient to drive an LLM regenerator with no additional human input. Three required fields:

```yaml
body:
  context_bundle:
    - kizo:web.todomvc                                # System parameters
    - kizo:web.todomvc.todo_management
    - kizo:web.todomvc.todo_management.todo_repository
    - kizo:web.todomvc.todo_management.todo_repository.todo_schema
    - kizo:web.todomvc.todo_management.todo_repository.spec.functional
    - kizo:web.todomvc.todo_management.todo_repository.spec.technical
    - kizo:web.todomvc.todo_management.todo_repository.spec.schema
    - kizo:web.todomvc.todo_management.todo_repository.spec.business_rule
    - kizo:web.todomvc.todo_management.todo_repository.spec.acceptance
    - pkg:generic/bun@1.1
    - pkg:npm/ulid
  outputs:
    - { path: src/db.ts, description: "Opens bun:sqlite, applies PRAGMAs, exports `db`." }
    - { path: src/repository.ts, description: "Exports class TodoRepository(db)..." }
    - { path: src/migrations/001_create_todos.sql, description: "DDL from todo_schema." }
    - { path: test/repository.test.ts, description: "bun:test suite covering spec.acceptance cases." }
  prompt_template: |
    You are generating the persistence layer of a TodoMVC clone.
    Read the bundled context files in order. Implement every method
    declared in spec.functional with the technical guidance in spec.technical
    and the rules in spec.business_rule. Produce the exact files in `outputs`.
    Then run the verifier and ensure it passes.

    HARD CONSTRAINTS:
    - Use bun:sqlite, NOT better-sqlite3 or sqlite3.
    - Apply PRAGMAs at connection open time, before any query.
    - Generate timestamps in JS (toISOString()), not in SQL.
  verifier:
    command: "bun test test/repository.test.ts && bun build --target bun src/repository.ts > /dev/null"
    expect: "all tests pass and build succeeds with no errors"
```

The `context_bundle` is the closure handed to the LLM as input — typically the Component itself, its child Interactions, all six spec leaves, and any external `pkg:` deps. The `outputs` enumerate every file the LLM MUST produce. The `prompt_template` ends with a HARD CONSTRAINTS list — these are the pitfalls the regenerator MUST avoid (often distilled from realization mistakes caught during the original authoring).

## 8. File-system layout

The convention validated by both sekkeis:

```
sekkei.yaml                              # root System node
nodes/
  systems/                               # one file per Sub-System (optional)
    register.yaml
    communication.yaml
    onlineordering.yaml
    reservations.yaml
  capabilities/                          # one file per Capability
    payment.yaml
    inventory.yaml
    ...
  components/                            # one file per Component OR a bundle
    payment_state_machine.yaml
    backend_components.yaml              # multi-doc bundle, `---` separators
    frontend_components.yaml
  interactions/
    sam_order_relay_fsm.yaml
    interactions.yaml                    # multi-doc bundle
  specs/
    by_component/                        # ONE FILE PER COMPONENT
      todo_repository_specs.yaml         # all 6 spec_kinds for this Component
      todo_filter_engine_specs.yaml
      ...
  dependencies/
    external_deps.yaml                   # PURLs and pinned versions
sekkei.lock                              # pinned (id, major, content_hash) per node
verify_sekkei.py                         # 6-gate verifier
```

Multi-doc YAML files use `---` separators and contain one node per document. A bundle file is appropriate when:

- The nodes are tightly coupled (e.g., the four Interactions of one Capability).
- Each node is small (under ~50 lines of body content).
- The bundle name describes the grouping (`backend_components.yaml`, `audit_pass_routes.yaml`).

When a Component grows enough to need more than ~3 spec leaves of substantive content, its specs MUST live in their own file at `nodes/specs/by_component/<component>_specs.yaml`. This is the "per-Component spec files" convention.

## 9. Verification gates

A sekkei is considered authored when it passes a 6-gate verifier. Both `kizo:food.fullservicerestaurant` and `kizo:web.todomvc` ship a `verify_sekkei.py` that implements these checks against the directory layout in §8. The gates:

| # | Gate                            | Checks                                                                                                                  |
|---|---------------------------------|-------------------------------------------------------------------------------------------------------------------------|
| 1 | Envelope                        | Every node has id, stratum, title, revision (major∈Y14.35, status∈enum), provenance (override_kind∈enum). Spec nodes have a valid spec_kind. |
| 2 | Stratum hierarchy               | Every `composes-of` edge respects the AllowedChildren table in §2.                                                      |
| 3 | Closure completeness            | Every `kizo:` target on `composes-of` / `depends-on` resolves to an authored node. External `pkg:`/`dep:`/`svc:`/`hw:` are the boundary. |
| 4 | Brief coverage                  | All required brief-named nodes (System, Capabilities, brief Components, brief Interactions) are present. List is sekkei-specific. |
| 5 | Spec coverage                   | Every Component has at least `functional`, `technical`, `acceptance`, and `prompt` spec leaves.                          |
| 6 | Spec quality                    | Every `spec.acceptance.body` has `deliverables` + `verifier`. Every `spec.prompt.body` has `context_bundle` + `outputs` + `verifier`. |

Gates 1, 2, 5, 6 are sekkei-agnostic (the verifier is portable). Gates 3 and 4 are parameterized by the sekkei (the brief list lives in the verifier script).

A failed gate 3 (dangling reference) is a *warning* in the BaanBaan/TodoMVC verifiers — closure can be partially authored during reverse-engineering and external services may be intentionally outside the closure. Gates 1, 2, 4, 5, 6 are *errors* and gate the A.0 acceptance.

## 10. Authoring conventions (not schema-enforced, but load-bearing)

These conventions emerged from the BaanBaan reverse-engineering and the TodoMVC validation. They are not enforced by the JSON Schema because they are stylistic rather than structural — but ignoring them produces sekkeis that pass gates 1-6 yet fail to regenerate working systems.

### 10.1 Brief naming

Components are named after WHAT THEY DO ("Payment State Machine", "Inventory Decrement", "Daily Closeout"), not where they live. The brief-named identity is durable across forks; the `kizo:` id encodes the fork-time placement only. A Component's `title` MUST be the brief name; the `id` may evolve as the fork is restructured.

### 10.2 Honest descriptions

Every Component body MUST state both what it OWNS and what it DOES NOT OWN. If a planned Component has no realization, mark `revision.status: in_work` and say so explicitly in `description` (`STATUS: PLANNED — NOT IMPLEMENTED`). Inventing slot-filler descriptions that don't match code is the failure mode; honest "this Component does not exist yet" is correct.

### 10.3 Read FSM states from source — never extrapolate

For any Interaction with `contract_kind: fsm`, the `states`, `actions`, and NAPs MUST be read directly from the realization file. Never extrapolate from the domain concept ("a payment FSM probably has authorize → capture → settle"). The realization-truth is what regeneration must reproduce. Inferring states from the abstract domain produced full rewrites of five files in the BaanBaan sekkei.

This rule is captured in the auto-memory file `feedback_read_fsm_before_describing.md`. A project that ignores it will pass the verifier but produce an FSM that does not match the code.

### 10.4 Acceptance specs are deliverables, not prose

The body of a `spec.acceptance` is a list of files-to-produce, the cases each file must cover, the runtime invariants, and a single executable verifier command. The legacy `inspection_assertions` shape is permitted for back-compat but is downgraded — new authoring uses deliverables.

### 10.5 Prompts are machine-runnable triples

The body of a `spec.prompt` is `(context_bundle, outputs, verifier)` plus a `prompt_template` that ends with a HARD CONSTRAINTS list. A regenerator MUST be able to execute the prompt with no additional human input.

### 10.6 Per-Component spec files

When a Component has more than ~3 spec leaves of substantive content, its specs live in their own file at `nodes/specs/by_component/<component>_specs.yaml`. Browsable, diffable, easy to dispatch to a focused regenerator.

### 10.7 YAML safety

Strings in flow-context mappings that contain `{`, `}`, `:` (other than the key separator), or `?` MUST be quoted. Common offenders:

- TypeScript-style optional params: `body: "{ title?: string, completed?: boolean }"`
- Inline object shapes in success descriptions: `success: "200 + { ok: true }"`
- URL paths with colons: `path: "/api/todos/:id"` (paths usually survive, but quoting is safer)

A YAML linter run as part of the verifier catches most of these; the verifier authored for both sekkeis runs `yaml.safe_load_all` on every file and emits a parse error rather than skipping.

## 11. Compatibility with the GLM essay

This document is a refinement of §C of the GLM specification, not a replacement. Differences are:

| Topic                          | Essay §C                              | This spec (v1.1)                                                                              |
|--------------------------------|---------------------------------------|------------------------------------------------------------------------------------------------|
| Strata                         | Strict 5-level hierarchy              | 5 levels + §C.1 amendment for System recursion (Sub-Systems)                                   |
| `spec_kind` location           | Inside `body`                         | TOP-LEVEL on spec nodes                                                                        |
| Acceptance spec body           | `inspection_assertions[]` (free prose) | `deliverables[]` + `runtime_invariants[]` + `verifier{command,expect}` (with legacy support)   |
| Prompt spec body               | (not specified separately)            | `context_bundle[]` + `outputs[]` + `prompt_template` + `verifier{command,expect}`              |
| Realization linking            | Implicit                              | Explicit `realization_file` / `realization_files` / `realization_notes` on Component / Interaction |
| FSM contract_definition        | Schematic ("SAM/FSM JSON")            | Concrete shape: `pc`, `pc0`, `states`, `actions`, `naps`, `reactors`, `invariants`             |
| Runtime enum                   | `in_process | sidecar | external_service` | + `in_browser` + `in_process_and_in_browser`                                                  |
| File-system layout             | (not specified)                       | Per-Component spec files in `nodes/specs/by_component/`; bundle files for tightly-coupled nodes |
| Verification                   | (mentioned as Puffin hooks)           | 6 explicit gates, with a portable `verify_sekkei.py` reference implementation                  |

The essay's seven GLM processes (Change Management, Variant Resolution, Where-Used, Effectivity, Drift Reconciliation, Reuse, Provenance) and the two-dimensional cache remain authoritative and out of scope here.
